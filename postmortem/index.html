Postmortem

What was the overall architecture of your system?

We used the sample project’s architecture as a guideline, though we started with an empty Android Studio project. We decided to split views and activities within the presentation folder to ease readability.

What went right in the development process?

Team members who struggled with issues and stories were transparent and asked for help which kept us from being stuck for a long time without making progress. We were able to complete all the user stories without delay for all three iterations.

What went wrong in the development process?

The COVID-19 outbreak made it hard to organize the group at first. During the whole process, we have been trying strategies to improve communication, especially at the beginning of the sprint, but we were not able to apply some of the ideas brought forward during retro1 because of social distancing.

What would you do differently, if you had the chance to start over?

We should have spent a bit more time testing different application templates. Our app is a bit slow because we start new activities on click instead of fragments. A better understanding of Android development in the beginning would have improved our app’s efficiency.

How large is the project (number of methods, classes, etc)? How much of this is (roughly) devoted to each major system component? And any other quantifiables (e.g. if you have a record of hours spent on tasks).

There are a total of 32 class files in our project. 12 are dedicated to the UI, 12 for the persistence layer, 4 for the business layer, and 4 dedicated to our object classes.

What took the most time? The least? Any surprises?

The most time would be the first iteration. This is because everyone was still getting used to Android Studio and version control, as well as understanding the overall architecture of the system. Once we were all on the same page things progressed smoothly.

Are there any particular design smells, or brilliant design decisions?

We intended to make our objects’ presentation different but a lack of foresight in what exactly we wanted to include ended up making them somewhat similar to each other. In hindsight, we could have built an “item” abstract class or interface and extend each object with common methods. Instead, many of our activity classes use a type conditional to call the method for each class that does essentially the same thing for different object types.

Are there any outstanding bugs?

The program does not display the UI as intended for certain devices.

Did any features work better than expected?

Features work as expected.

Are you using any technologies other than what was required (e.g. JMock, GUI builders, etc.)?

We are not.

Are you using any specific techniques covered in the course (TDD, pair programming, scrums, etc)?

We did not use any specific techniques discussed in class during our development process.

How did the project change from your initial (iteration 0) vision or stories, or did it work out as predicted?

The different objects (notes, flashcards, reminders) are not as fleshed out as initially predicted but, overall, the project is very close to the initial vision.

What did you learn about team or large project development? What will you start doing, keep doing, or stop doing next time?

That communication and ensuring everyone is on the same page with regards to the project’s vision and architecture is important to the overall success of the development process. Start: Being more clear with regards to our vision of the finished product. Keep: Retrospectives at the end of an iteration or big feature. Stop: Waiting until the last minute to start working on features.

Can you draw any conclusions from what you’ve done?

I believe that after completing our project we have all gained a large amount of valuable experience in working in groups, developing large projects, and using agile development practices to create software applications.
